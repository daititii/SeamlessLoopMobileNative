# PC 数据库同步与歌单创建极致优化总方案

## 1. 现状痛点与原因分析

目前项目在多选歌曲、创建歌单以及同步 PC 端循环数据时，面临以下几个严重问题：

*   **创建歌单极其耗时/卡顿**：由于在创建歌单绑定了一连串读取外部文件（如 DB 或用 C++ 解析音频）的过程，导致点击 UI 操作后产生不可预知的漫长等待。
*   **并发冲突与死锁重叠**：如果在前一个扫描还没有完成时，又立刻创建了下一个歌单，会导致多个协程抢占式读取同一组底层文件，造成互相踩踏。
*   **同名歌曲匹配错乱与幽灵数据**：目前 `PcDatabaseImporter.kt` 里面强行把 PC 的“总采样数”当成了手机的毫秒“duration”塞进了数据库；同时，盲目的 `insert` 导致数据库里出现大量没有真实文件路径（FilePath 为空）的“幽灵数据”。
*   **无法全选或多选添加到歌单**：底层数据处理太重，UI 层被拖累，加上老版本的“同步文件夹操作”剥离，导致目前无法顺畅跑通建单逻辑。

## 2. 核心架构重构思路：【完全解耦，前重后轻】

参考主流高级音乐播放器的设计理念，我们将彻底把**“繁重的底层数据解包与同步”**与**“轻快的用户界面操作”**分离开来：

### 2.1 抛弃强行同步插入，采用“精准定位更新 (Update)”
PC 端产生的 DB 文件，仅仅是一本“参考答案字典”。
我们在同步时，绝不再往手机本地歌单里凭空乱造不存在的歌曲幽灵。而是去手机本地已经扫出来的**现有曲库**里找，找到有缘人，再把“循环点”数据精准更新（Update）给它。

### 2.2 彻底解决“同名”难题：(44.1k / 48k) 双重时长交叉校验法
为了在不需要知道文件路径的前提下精准识别：
利用 `文件名 (FileName)` 锁定候选人。
再利用 PC 端的 `TotalSamples` 倒推出这首音乐在 **44100Hz** 和 **48000Hz** 下可能的标准毫秒时长。
跟手机曲库里这首同名歌曲的 `duration` 进行比对（容许 ±1500ms 误差）。全对上，即是真身！

### 2.3 歌单 UI 瞬间响应（毫秒级）
完成了上面的数据同步铺垫后，手机库里的歌曲已经被赋予了循环点。
所有针对歌单的“创建”、“全选”、“添加”，都不再涉及任何一次底层扫描！它们变成了纯内存与 SQLite 关系表的秒级读写。

## 3. 详细执行四个步骤（重构路线图）

### 👣 第一步：解除紧致的实体表约束 (Entity)
**修改文件：** `Song.kt`
*   移除在数据库索引上对 `["FileName", "duration"]` 的 `unique = true` 强制约束。
*   允许同名、同近时长的歌曲合理共存，避免约束冲突导致的程序崩溃。

### 👣 第二步：增加“只打补丁”的数据库方法 (Dao)
**修改文件：** `SongDao.kt`
*   新增专门用来更新循环数据的 `@Query` 方法：
    ```kotlin
    @Query("UPDATE LoopPoints SET LoopStart = :start, LoopEnd = :end, TotalSamples = :total WHERE Id = :songId")
    suspend fun updateLoopPoints(songId: Long, start: Long, end: Long, total: Long)
    ```

### 👣 第三步：彻底重写 PC 数据库搬运工 (核心算法)
**修改文件：** `PcDatabaseImporter.kt`
*   将原有的“粗暴读取+盲目 Insert”彻底废弃。
*   植入全新的聪明验证法：
    1. 提取 PC DB 的一条数据；
    2. 在手机现存曲库 `List<Song>` 中寻找同名候选人；
    3. 用 `TotalSamples / 44.1` 或 `48.0` 计算预估毫秒并校验；
    4. 对通过测试的真命天子，下发 `updateLoopPoints` 补丁更新。
*   执行完后，再调用 `scanLibrary` 通知 UI 刷新。

### 👣 第四步：彻底重构打通“创建歌单”UI与相关逻辑
**修改文件：** `SelectionController.kt` / `MainViewModel.kt` 等（根据后续实现情况判定）
*   修复歌单不能“全选”的基础 Bug。
*   将“创建歌单/添加歌曲”逻辑改写为纯粹的调用 `PlaylistDao.insert...` 的方式。
*   彻底摘掉跟“扫描文件”有关的前台包袱。
