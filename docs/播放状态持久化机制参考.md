# 播放状态持久化与状态恢复机制参考

本文档整理自对 `APlayer` 项目源码的分析，旨在为 `SeamlessLoopMobile` 提供更加稳健的“记住上次播放状态”的实现思路。

## 1. 核心思想：双重存储策略 (双线作战)

在处理大量播放列表数据和轻量级播放状态时，不应将所有数据混合存放在单一的持久化介质（如 `SharedPreferences`）中。

- **大量数据（播放队列）存数据库**：利用底层的 Room 或 SQLite 专门保存当前或上次使用的整个歌曲列表。这样即便播放列表包含数千首歌曲，也不会像 JSON 序列化那样导致 `SharedPreferences` 文件体积膨胀、读写缓慢或出现内存溢出风险。
- **轻量状态（游标与进度）存 SharedPreferences**：仅将“最后播放的歌曲标识（ID或路径）”作为游标（Cursor），以及“最后的毫秒进度”作为轻量级状态存入偏好设置中。

## 2. 精准的复原逻辑 (合并启动)

在应用或后台服务重启时，通过解耦的数据源进行状态还原：

1. **从数据库抽取队列**：服务启动时，优先从数据库（如 `PlayQueueRepository`）中完整抽取上次持久化的播放队伍。
2. **从偏好设置提取书签**：并行或随后从 `SharedPreferences` 中提取 `lastSong` 和 `lastProgress`。
3. **定位排头兵**：在抽取出的队伍中遍历，核对歌曲的 ID 或路径，找到上次播放歌曲所处的精确索引（也就是在队列中的 `pos` 位置）。
4. **一键载入引擎**：将完整的队列、当前的索引位置以及毫秒进度合并，一次性传递给底层播放引擎（如 `ExoPlayer` 或 `NativeAudio`）进行就绪准备。

## 3. “死皮赖脸”的进度记录机制 (轮询保存)

通常的进度保存依赖于“生命周期事件”（如暂停、切歌、应用退到后台）。但这在遇到系统强杀（OOM 机制）、应用崩溃或突然断电时，容易丢失大段的播放进度。

**更具韧性的做法**：
- 启动一个专用的协程轮询器（如定时任务 `progressJob`）。
- 只要引擎处于播放状态，就每隔固定的时间（例如 5 秒或 10 秒）主动获取当前真实进度，并覆盖写入 `SharedPreferences`。
- 这样，即使应用遭遇意外强杀，用户重启应用时最多也只会丢失几秒钟的进度，极大地提升了体验的连贯性。

## 4. 对 SeamlessLoopMobile 的启示

如果我们希望进一步优化 `SeamlessLoopMobile` 的记忆系统，可以参考以下改造路线：
- **剥离列表**：将当前的 `SettingsManager.currentPlaylistPaths` 废弃，改为在 `Room` 数据库中新建一张表专门存储每次变更后的实际播放队列集。
- **进度策略调整**：根据 cpu 大人的最新要求，**不再需要记录毫秒级进度**，应用重启后默认从所选歌曲的开头播放即可。这也使得系统变得更加轻量，我们无需实现“死皮赖脸”的进度定时轮询机制喵！
