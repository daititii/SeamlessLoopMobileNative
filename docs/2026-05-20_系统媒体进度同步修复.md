# 2026-05-20 系统媒体进度同步修复

## 背景

本次修复针对一次循环后播放页进度条与系统通知栏/锁屏媒体进度不同步的问题：

- 普通单曲循环：播放页已经回到循环起点附近，系统通知栏进度仍停留在结尾附近。
- AB 模式：B 段循环后也会出现同类现象。

播放页显示的是 native 当前可听位置；系统通知栏/锁屏依赖 `MediaSession` 的 `PlaybackStateCompat.position`。两条链路在 loop handover 后的刷新时机不同，导致系统侧进度可能短暂或持续落在旧位置。

## 根因

`EVENT_LOOP_JUMP` 表示 native 解码侧完成了一次 loop handover，但它不是“用户已经听到回绕位置”的权威时刻。

旧逻辑在收到 `EVENT_LOOP_JUMP` 后立即调用 `PlaybackManager.updatePosition()`，会让系统媒体进度刷新时机依赖解码切换事件。这个事件和实际音频输出、`mCurrentReadFrame` 可听位置之间存在缓冲和线程时序差异，因此系统栏可能与播放页出现错位。

## 修改内容

### 1. 明确 `EVENT_LOOP_JUMP` 语义

`PlaybackManager` 不再在 `EVENT_LOOP_JUMP` 中立即发布 `MediaSession` 进度。该事件只保留日志与注释，用来表示 native 解码切换完成。

`AudioEngine.cpp` 补充注释，明确 `EVENT_LOOP_JUMP` 不是系统进度应立即回绕的信号。

### 2. 新增系统进度同步器

新增 `SystemMediaProgressSyncController`：

- 仅在 `AudioPlayState.PLAYING` 时启动。
- 默认每 250ms 调用一次同步 tick。
- `PAUSED`、`IDLE`、`ERROR` 和 `dispose()` 时可靠停止。
- 每次 tick 都调用 `PlaybackManager.updatePosition()`，由它采样当下 native 播放位置并发布到 `MediaSession`。

这样系统通知栏/锁屏进度的唯一真源变为“采样时刻读取到的 native 当前播放位置”，而不是 loop handover 事件。

### 3. 由 `PlaybackService` 持有同步生命周期

同步 job 由 `PlaybackService` 创建并监听 `PlaybackManager.state`：

- 后台播放和锁屏场景不依赖 Activity 是否连接。
- `MediaControlManager` 的 UI 侧轮询保持不变。
- 不把 JNI 或 MediaSession 调用挪到 native 音频回调线程。
- steady-state 刷新节奏保持在 250ms，不额外扩大轮询负担。

### 4. 帧到毫秒换算集中化

新增 `framesToPlaybackPositionMs(positionFrames, sampleRate)`，统一处理 native 帧位置到 `MediaSession` 毫秒位置的转换，并在采样率异常时兜底为 44100。

## 测试覆盖

新增 `SystemMediaProgressSyncControllerTest`，覆盖：

- 只有 `PLAYING` 状态启动同步 job。
- 重复进入 `PLAYING` 不会创建重复 job。
- `ERROR` 和 `dispose()` 后不再继续 tick。
- 44100 / 48000 采样率下帧到毫秒换算正确。
- loop 后发布的是 tick 时采样到的 native 位置，而不是由 `EVENT_LOOP_JUMP` 推导出的提前回绕位置。

## 手测建议

1. 普通单曲循环：把 loop 点放在尾部附近，连续观察 3 次以上回环。
2. AB 模式：覆盖 A -> B 过渡和 B -> B 循环。
3. 验证播放页听感不变，系统通知栏/锁屏进度在实际回绕后的一个 250ms tick 内重新对齐。
4. 观察是否有新增爆音、卡顿、日志刷屏或明显耗电上升。

## 后续注意

如果以后追求系统栏在 loop 瞬间视觉零延迟回绕，需要 native 侧提供更精确的 audible-loop 标记，而不是复用 `EVENT_LOOP_JUMP`。当前方案选择的是稳定和低成本的 250ms service-side 采样同步。
