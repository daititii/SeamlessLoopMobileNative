# 移动端音频引擎流式重构工作日志

**日期**：2026-02-17  
**记录人**：莱芙·泽诺 (Lev Zenith)  
**项目**：SeamlessLoopMobile  

## 1. 背景与问题
在将桌面端的无缝循环播放器移植到 Android 平台时，初期采用了 **全量预加载 (Full Preloading)** 方案。该方案存在以下严重缺陷：
- **启动慢**：加载长音频（如 10 分钟以上的曲目）需要等待数秒甚至更久，直到整个文件解码完成。
- **内存占用高**：解码后的 PCM 数据全部驻留内存，导致 App 容易被系统杀掉 (OOM 风险)。
- **交互卡顿**：缺乏流式支持，无法实现进度条拖拽 (Seek) 和动态循环点调整。
- **崩溃与杂音**：多线程并发操作缺乏保护，导致 `SIGBUS` 崩溃和音频 Underrun 杂音。

## 2. 实施变动 (Refactoring)

### 2.1 核心解码器重构 (`AudioDecoder`)
- **从函数到对象**：将 `AudioDecoder` 从单一的静态工具函数重构为 **状态化类**。
- **流式读取**：实现了 `readSamples()` 接口，内部维护小容量 `mInternalBuffer`，支持分块解码，按需供给。
- **精准跳转**：封装了 `AMediaExtractor_seekTo` 和 `AMediaCodec_flush`，实现了 `seekToFrame()`，为进度条和循环跳转打下基础。
- **状态判断**：增加了 `isFinished()` 接口，准确区分 "暂时没数据 (Underrun)" 和 "真正结束 (EOS)"。

### 2.2 音频引擎逻辑更新 (`AudioEngine`)
- **移除巨型缓冲**：删除了 `std::vector<float> mAudioBuffer`，不再持有整首歌曲数据。
- **实时并流**：在 Oboe 的 `onAudioReady` 回调中，直接调用 `mDecoder->readSamples()` 获取数据，实现了 **边解边播**。
- **循环逻辑迁移**：复刻了桌面端 `LoopStream.cs` 的核心算法。实时监测当前帧位置，一旦触及 `LoopEnd`，立即调用 `decoder->seekTo(LoopStart)`，实现无缝循环。
- **线程安全**：引入 `std::mutex` 和 `std::atomic`，彻底解决了 JNI 层多线程资源竞争导致的崩溃问题。

### 2.3 性能优化
- **秒开体验**：`loadAudioSource` 仅解析元数据，不再进行耗时解码，点击即播。
- **低内存占用**：通过流式处理，无论音频文件多大，内存占用始终维持在极低水平（仅取决于缓冲区大小）。

## 3. 遇到的挑战与修复
- **死循环 bug**：初期重构时，因未正确处理解码器 EOS 状态，导致 `AudioEngine` 误以为播放结束并反复触发 Loop Seek，引发日志刷屏。
  - **修复**：在 Seek 前强制检查 `mDecoder->isFinished()`，确保仅在真正结束时才跳转。
- **长整型格式化警告**：修复了 C++ `LOGD` 中 `int64_t` 格式化字符串的编译器警告。

## 4. 后续计划
- **UI 交互**：实现进度条 (SeekBar) 显示与拖拽功能。
- **A-B 循环控制**：在界面添加按钮，允许用户实时设置 Loop Start / Loop End。
- **JNI 接口扩展**：暴露 `getCurrentPosition`, `getDuration`, `seekTo` 等接口给 Kotlin 层。
