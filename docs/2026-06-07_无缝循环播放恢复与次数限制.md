# 2026-06-07 无缝循环播放恢复与次数限制

## 背景

本次集中处理三类播放体验问题：

1. 无缝循环开启时，用户把播放进度拖到循环终点之后，播放位置应回到循环起点，但实际会落在循环起点之后的偏移位置。
2. 暂停播放后清除通知栏，播放器 native engine 被销毁，后续点击播放无法直接恢复，需要切歌才能重新播放。
3. 新增“单曲无缝循环次数上限”设置：默认 0 表示无限循环；大于 0 时，当前歌曲完成指定次数的无缝回绕后触发调度。

## 修复与实现

### 1. seek 越过 loopEnd 的回弹

- 修改 `AudioEngine::seekTo()`：
  - 负数 seek 钳制到 0。
  - 无缝循环开启且目标位置 `>= loopEnd` 时，将 seek 目标改为 `loopStart`。
- 保留“手动 seek 到 loopStart 之前”的能力，用于试听前奏/引入段。

### 2. 清除暂停通知后的播放恢复

- `PlaybackManager.resume()` 在 `currentSong == null` 时不再把状态误置为 PLAYING，而是回到 IDLE 并通知状态回调。
- `PlaybackService.onPlay()` 改为 `resumeOrRestorePlayback()`：
  - engine 仍存在且当前歌曲有效时正常 resume。
  - engine 已销毁时，通过 `SettingsManager.lastSongPath` 和 `lastPosition` 重新加载上一首。
  - `lastPosition` 以帧为单位保存和恢复。
  - AB 模式下按 A+B 虚拟时间轴估算恢复位置上限。

### 3. 全屏播放页进度显示保护

- 全屏播放页的 `PlaybackProgressBar` 继续直接轮询 `NativeAudio`，保持拖动低延迟和即时反馈。
- 当暂停后清除通知导致 native engine 被销毁时，`NativeAudio.getDuration()` / `getCurrentPosition()` 会返回 0。此时进度条不再用 0 覆盖已有 UI 进度。
- 若刚进入全屏页尚未拿到 native 进度，则使用通知栏同源的 `PlaybackState.position` / `MediaControlManager.currentPosition` 作为初始兜底。
- 一旦 native 重新提供非 0 进度，进度条回到 native 进度源。

### 4. 单曲无缝循环次数上限

- `SettingsManager` 新增 `seamlessLoopCountLimit`：
  - 默认值：0。
  - 上限：`MAX_SEAMLESS_LOOP_COUNT_LIMIT = 9999`。
  - 0 表示无限循环，永不因次数限制自动调度。
- 设置侧边栏新增“无缝循环行为”卡片：
  - 输入框仅允许保存非负整数。
  - 空输入、非数字字符、整数溢出、超出上限都会显示明确错误，并禁用保存按钮。
- `PlaybackManager` 监听 native `EVENT_LOOP_JUMP`：
  - 该事件代表解码线程已经完成 loopEnd→loopStart 的逻辑跳转。
  - 计数不是在事件到达时立即增加，而是在服务端进度同步采样到实际播放位置回绕后增加，避免系统通知进度还没可感知回绕就提前切歌。
  - 手动 seek 会清空 pending loop jump，避免误计数。
- 达到次数上限后的行为：
  - LIST_LOOP / SHUFFLE：切换下一首。
  - SINGLE_LOOP：重新播放当前歌曲并重置计数，不切换下一首。

用户实机验证通过：

- 播放页暂停后清除通知，UI 进度不再被错误归零。
- 清除通知后再次播放，能从正确位置恢复。
- 一级/二级播放页切换后进度显示正常。
- 无缝循环次数上限功能测试通过。
- 单曲循环模式下达到次数上限不会切到下一首。
